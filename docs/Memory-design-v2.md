# Система памʼяті проєкту: кураторський markdown + векторний індекс (v2)

> v2 — еволюція концепції після розвідки. Ключова зміна проти v1: claude-mem НЕ використовується як двигун (він архітектурно під протилежне — авто-захоплення сесій). Будуємо власний тонкий шар. Векторний пошук — основний механізм.

## Область видимості / джерело правди

Основний принцип — усі `.md` файли памʼяті живуть у репозиторії на рівні проєкту і є ЄДИНИМ джерелом правди.

Формат файлів — нативний формат памʼяті Claude Code (тонкий `MEMORY.md`-індекс + директорія логів + topic-файли), дзеркалений у `.claude` директорії проєкту для сумісності з системним форматом памʼяті.

Векторний індекс — це локальний, одноразовий, відтворюваний ПОХІДНИЙ артефакт. Він не в git. Його завжди можна видалити й перебудувати з `.md`.

## Векторна база

Основний принцип — одна СУБД у вигляді єдиного файлу SQLite, яка в собі тримає одночасно: векторний індекс, повнотекстовий індекс і власний стан змін (хеші).

* Векторний (семантичний) пошук — ОСНОВНИЙ механізм отримання релевантних знань.
* BM225 / повнотекстовий — ДОДАТКОВИЙ, вторинний запобіжник на випадок точного збігу по слову/терміну. Не основний шлях.
* Без сервера, без Docker, без демона, без зовнішньої БД. Один локальний файл, gitignored, відтворюваний.

Розподіл області видимості — по проєктах, а всередині проєкту: загальна памʼять проєкту + агентні блоки памʼяті по-агентно (для агентів, представлених у проєкті).

## Ембединг-модель

Основний принцип — модель ембедингу локальна, і її тримає завантаженою легкий локальний процес-резидент: ембединг обчислюється на місці, без повторного завантаження моделі на кожен запит.

* Лише локально (loopback), нічого мережевого назовні, нічого в git, нічого встановлювати окремо; резидент авто-керований — піднімається за потребою, звільняє памʼять у простої, один на машину.
* Модель завантажується тільки явним прогрівом, ніколи неявно; резидент лише тримає вже наявну модель.
* Доступ до ембедингу — тонкі клієнти резидента; за його відсутності читання деградує мʼяко, без блокування.

## Запис або редагування памʼяті як файлів

Агенти використовують нативні інструменти оновлення memory та/або інструменти редагування `.md` файлів (у ролі окремих від нативної memory файлів із памʼяттю).

Коли трігериться:
* хуки на стоп сесії,
* прописані рулзи/інструкції, які зобовʼязують підтримувати памʼять в актуальному стані, та/або у випадках значних змін проєкту, після комітів і т.д.

## Синхронізація `.md` → векторний індекс (концепція sync-скрипта)

Основний принцип — синхронізація ОДНОНАПРАВЛЕНА: `.md` → індекс. Запис завжди йде у `.md`; індекс ніколи не редагується напряму, він лише похідний.

Коли трігериться:
* хук на оновлення memory-файлу — інкрементально, лише цей файл,
* хук на старт сесії — повний reconcile (бо файли могли змінитися через git pull / clone / branch switch, а не агентом).

Концепція скрипта:
* обходить усі наявні `.md` файли, що слугують памʼяттю,
* рахує хеш кожного файлу і порівнює з попереднім станом; хеші зберігаються в ТІЙ САМІЙ базі (не в окремому файлі — щоб стан не розходився з реальністю),
* переіндексовує у векторну базу ЛИШЕ ті файли, що змінилися,
* одиниця памʼяті — секція файлу зі стабільним детермінованим ідентифікатором (однаковим на всіх машинах),
* ОБОВʼЯЗКОВО видаляє з індексу те, чого вже немає у файлах (видалені або перейменовані) — prune. Без цього накопичується мертвий баласт.

Властивості: ідемпотентність (повторний запуск без змін — нічого не робить), детермінізм, повна відтворюваність (видалив базу → переіндексував → той самий стан).

## Читання памʼяті (пошук)

Основний принцип — отримання релевантних знань із загальної памʼяті проєкту та/або власних знань-приміток агента (як особистого скоупу) через семантичний векторний пошук; BM225 — додаткове підсилення на точні терміни; коли працюють обидва — результати зливаються, кращий сигнал виграє сам.

Дві точки входу, але ОДИН пошуковий рушій під ними (запис і читання — одна когерентна система, не дві):
* MCP-інструмент — агент САМ у довільний момент часу йде й спеціально шукає за своїми запитами, так як вважає за потрібне, з потрібним скоупом (загальне / свій агентний блок). Це ядро.
* Автоматичне підкидання релевантного в контекст — через хуки (старт сесії та/або перед обдумуваннями/після введення користувача). Опційно; точна механіка та доцільність — окреме рішення на потім.
* Захист: порожній або сміттєвий запит / слабкі збіги не підкидаються в контекст, щоб його не засмічувати.

# Пайплайни реалізації

Принцип реалізації — компонуємо тонкі готові частини, але векторне сховище НАШЕ (sqlite-vec), схему й стан контролюємо ми. Складний шматок (розбиття на чанки) не винаходимо — беремо `semantic-text-splitter`. Решта поверх sqlite-vec — тонкий власний код.

## Пайплайн запису (`.md` → векторний індекс)

1. Наш Python-код обходить файли памʼяті, рахує sha256, порівнює зі станом у базі — визначає що додалося / змінилося / зникло.
2. Для змінених файлів `semantic-text-splitter` ріже їх на чанки по заголовках (малий файл = один цілий чанк).
3. Локальна ембединг-модель перетворює чанки на вектори.
4. Наш Python-код пише чанки + вектори + метадані (project, agent, шлях, стабільний id) у НАШ sqlite-vec; видаляє з індексу чанки файлів, яких уже немає — prune.

Тригери: оновлення memory-файлу — інкрементально, лише цей файл; старт сесії — повний reconcile.

## Пайплайн читання (запит → релевантні секції)

1. Наш Python-код формує запит: або запит агента (через MCP-інструмент), або текст користувача (через хук).
2. sqlite-vec робить векторний (семантичний) пошук — ОСНОВНЕ.
3. Опційно BM225 / FTS5 по тих самих чанках — ДОДАТКОВЕ; результати зливаються простим RRF.
4. Фільтр по метаданих (скоуп project / agent) застосовується в самому запиті.
5. Наш Python-код повертає топ-N релевантних секцій, з гейтом проти порожніх / слабких збігів.

Точки входу: MCP-інструмент (агент сам шукає коли хоче) — ядро; авто-підкидання через хук — опційно, рішення на потім.

# Ролі агентів

## Claude Code консоль

Роль: тімлід, сам нічого не змінює і не створює, головна задача — керувати командою агентів, прописаних у проєкті, запускати своїх тімейт-агентів як команду, працювати й комунікувати з ними.

Памʼять: отримує результати зі спільної памʼяті проєкту, записує памʼять як загальну памʼять проєкту.

## Член команди агентів з проекту

Роль: комунікувати з тімлідом та іншими агентами, виконувати свою роботу у чітко окресленому домені (агент девелопер, агент тестувальник, агент аналітик, ...).

Памʼять: отримує результати зі спільної памʼяті проєкту, можливо також зі своєї памʼяті як сутності (але все одно на рівні проєкту), записує памʼять як загальну памʼять проєкту та/або памʼять на рівні свого агента.

# Антипатерни, яких свідомо уникаємо (уроки розвідки)

* Немає prune → стейл при ренеймі/видаленні файлів. (Головна причина провалу попередньої спроби в іншому проєкті.)
* Хуки в нестандартному файлі, який Claude Code не читає → нічого не тригериться. Хуки — лише у git-tracked налаштуваннях проєкту, без машинно-залежних абсолютних шляхів.
* Розрив «запис ≠ читання» (різні підсистеми, що не зʼєдналися) → пишемо й читаємо через один спільний рушій.
* Окремий manifest хешів поряд із базою → розсинхрон стану з реальністю. Хеші живуть у самій базі.
* Важка або зовнішня інфра (окремий сервер, Postgres, Docker, мережева служба, зовнішня БД) → суперечить принципу «локальне, одноразове, їде з git clone». Не вводимо.

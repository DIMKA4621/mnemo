# Caching

Redis — кеш і брокер черг. Кешування навмисно консервативне: краще
зайвий запит у БД, ніж віддати клієнту застарілі дані.

## Cache keys & TTL
Ключі — з префіксом сервісу й версією схеми: `core:v3:user:{id}`.
TTL за замовчуванням 5 хв; для рідкозмінних довідників — до 1 год.
Без TTL не кешуємо нічого взагалі.

## Invalidation
Інвалідація — за подією на запис: сервіс, що змінює сутність, явно
видаляє відповідні ключі. Не покладаємось лише на TTL для критичних
даних. Версія в ключі дозволяє «масово протермінувати» весь namespace
зміною версії, без сканування Redis.

## Cache stampede
При протермінуванні гарячого ключа багато запитів б'ють у БД одночасно
(stampede). Захист: probabilistic early expiration + короткий
розподілений лок на перерахунок. Один запит перераховує, решта віддають
ще валідне старе значення, поки не готове нове.

## Hot keys & memory
Стежимо за гарячими ключами через `redis-cli --hotkeys`. Воркер тримає
обмежений LRU поверх Redis для найгарячіших — саме його некерований
ріст колись спричинив memory leak, тому розмір LRU тепер жорстко
обмежений.

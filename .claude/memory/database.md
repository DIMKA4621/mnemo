# Database

PostgreSQL 16 — основне сховище. Усі рішення по схемі, міграціях,
пулінгу, реплікації, таймаутах і бекапах тримаємо тут.

## Schema & migrations
Міграції — через Alembic, по одному revision на PR. Кожна міграція має
бути **backward-compatible**: спершу деплоїться код, що працює і зі старою,
і з новою схемою, потім — сама міграція. Drop колонок робимо окремим
релізом через тиждень, коли впевнені, що старий код не використовується.
Ніколи не редагуємо вже змерджену міграцію — лише нова revision поверх.
Бекфіл великих таблиць — батчами по кілька тисяч рядків із паузою, щоб
не роздути WAL і не зловити реплікаційний лаг.

## Connection pooling
Використовуємо PgBouncer у transaction-режимі. Пул на сервіс обмежений
20 конектами. Довгі транзакції заборонені — вони тримають конект і
вичерпують пул; важкі операції виносимо у Worker через чергу. Кожен
сервіс має окремий пул-юзер, щоб можна було ізолювати й діагностувати
навантаження по сервісах окремо.

## Query timeouts
На рівні ролі застосунку виставлено `statement_timeout = 5s` і
`idle_in_transaction_session_timeout = 10s`. Запит, що не вклався, краще
впаде з помилкою (її видно в алертах), ніж висітиме й триматиме лок.
Аналітичні важкі запити йдуть окремою роллю з більшим таймаутом і тільки
на репліку.

## Read replicas
Одна синхронна репліка для failover, одна асинхронна для читання звітів.
Застосунок не читає з репліки за замовчуванням — лише явно позначені
read-only запити, бо є реплікаційний лаг (зазвичай <1s, але не нуль).
Після запису не читаємо ту саму сутність із репліки в межах того ж
запиту — це класична причина «зник мій щойно збережений запис».

## Indexing strategy
Індекси додаємо за реальними планами (EXPLAIN ANALYZE), не «про всяк
випадок». Створення індексів на проді — завжди `CREATE INDEX
CONCURRENTLY`, щоб не лочити таблицю. Композитні індекси — у порядку
від найселективнішої колонки. Невикористані індекси видаляємо: вони
сповільнюють запис і їдять місце.

## Backups & restore
Базу бекапимо `pg_dump` щодня + безперервний WAL-архів (PITR).
Відновлення тестуємо щомісяця на окремому інстансі — бекап, який не
перевіряли відновленням, вважається відсутнім. RPO 5 хв, RTO 1 год.
Перед ризикованою міграцією — позачерговий бекап і зафіксована точка
відновлення.
